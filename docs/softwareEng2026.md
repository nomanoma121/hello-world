# ソフトウェア工学 まとめ

## ソフトウェアとは

ソフトウェアは、プログラムだけでなく、データ構造や操作方法の説明も含む。

ハードウェアのように物理的には劣化しないが、OS・ライブラリ・要求・利用環境が変わることで、相対的に「悪化」する。  
そのため、ソフトウェアは作って終わりではなく、変更・保守され続けるものとして考える必要がある。

## ソフトウェア工学とは

ソフトウェア工学は、ソフトウェアの開発・運用・保守に対して、体系的な方法を適用する分野である。

目的は、以下のバランスを取ること。

- 品質
- コスト
- 納期

単にコードを書くのではなく、長期運用、チーム開発、変更への対応、品質管理まで含めて考える点が重要である。

## 基本的な考え方

ソフトウェア開発では、以下の流れが基本になる。

1. 問題を理解する
2. 解決策を計画する
3. 実装する
4. 正しく動くか確認する

また、よい開発では以下が重視される。

- ユーザーに価値を提供する
- できるだけシンプルにする
- 明確な目的を持つ
- 他人が理解できる設計・実装・ドキュメントにする
- 将来の変更に備える

## ソフトウェア工学が必要な理由

現代では、社会やサービスの多くがソフトウェアに依存している。  
そのため、大規模で複雑なソフトウェアを、効率よく高品質に開発・保守する必要がある。

また、IT人材不足や社会インフラとしての責任、ユーザー要求の変化にも対応する必要がある。

## プログラミングとの違い

プログラミングは主にコードを書いて動くものを作ること。

一方、ソフトウェア工学では以下も考える。

- 保守
- 長期運用
- チーム開発
- テスト
- ドキュメント
- 依存ライブラリやOSの更新
- ユーザー要求の変化

つまり、ソフトウェア工学は「長く使われるソフトウェアを安全に作り続けるための考え方」である。

## Hyrumの法則

十分な数のユーザーがいるAPIでは、仕様として保証していない挙動であっても、誰かが依存する可能性がある。

そのため、開発者が「仕様ではない」と思っている挙動でも、変更すると影響が出ることがある。  
リリースや仕様変更では、観察可能な挙動への依存に注意する必要がある。

## ソフトウェアライフサイクル

ソフトウェアには、誕生から廃止までの流れがある。

1. ニーズの発生
2. 企画・計画
3. 要件定義
4. 設計
5. 制作・実装
6. テスト・デバッグ
7. 運用・保守
8. 廃止・リニューアル

## 要件定義

要件定義では、どのようなソフトウェアが必要かを明確にする。

主な内容は以下。

- システムの目的
- 必要な機能
- システム構成
- 目標性能
- 他システムとの連携
- 制限事項
- 拡張性
- 開発スケジュール
- 開発体制
- 納品物

重要なのは、実現する機能だけでなく、実現しない機能も明確にすること。  
要件定義書は、プロジェクト関係者の共通認識として使われる。

## 設計・制作

設計では、要件定義の内容をもとに、実際にコーディングできるレベルまで仕様を落とし込む。

制作では、設計書に従って実装を行う。  
開発は自社で行う場合もあれば、外部に委託する場合もある。

外部委託では、ソースコードを納品物に含めるかどうかが、後の保守に関わる。

## テストとデバッグ

バグのないソフトウェアは基本的に存在しないと考えるべきである。

そのため、事前にテストを行い、バグを早期に発見・修正することが重要。  
ただし、すべてを完全にテストすることは不可能なので、どこをどの程度テストするかを考える必要がある。

## 運用・保守とドキュメント

ソフトウェアはリリース後も運用・保守される。

運用時には、開発者がすでにプロジェクトから離れている場合もある。  
そのため、ドキュメントは複数人で開発・保守するための重要なコミュニケーション手段になる。

## プロジェクトとバックキャスティング

プロジェクトとは、有期性と独自性を持つ仕事である。

- 有期性: 必ず終わりがある
- 独自性: 固有の目的がある

プロジェクトでは、現在から積み上げるだけでなく、目標から逆算して必要な作業を考えるバックキャスティングが重要になる。

## ソフトウェアの評価

ソフトウェアは、以下の観点から評価される。

- 物量
- 実行性能
- 開発工数
- 品質
- 使い勝手

また、品質・コスト・納期のうち何を優先するかは、システムの性質によって変わる。

例:

- 人命に関わるシステム: 品質優先
- 公共関連システム: 費用優先
- イベント関連システム: 納期優先

## 開発プロセス

### ウォーターフォール型

工程を順番に進める開発手法。  
進捗管理がしやすく成果物も明確だが、後工程に問題が集中しやすい。

### スパイラルモデル

小さなフェーズに分け、プロトタイプを作りながら進める手法。  
フィードバックを得やすいが、プロトタイプ作成に工数がかかることもある。

### 反復型開発

機能ごとに分割し、反復単位で開発する手法。  
顧客要求を取り入れやすいが、管理の手間が増える。

### アジャイル開発

変化に対応しながら、短いサイクルで動くソフトウェアを提供する手法。

重視するもの:

- プロセスやツールより対話
- 詳細なドキュメントより動くソフトウェア
- 契約交渉より顧客との協調
- 計画遵守より変化への対応

## 開発手法の選び方

開発手法には一長一短があり、どれが最適かはプロジェクトによって異なる。

考慮すべきリスクには、以下がある。

- 人的リスク
- 技術的リスク
- 規模の変化
- ニーズの変化
- 社会的要因
- 予測不能なトラブル

実際には、複数の手法を組み合わせることも多い。

## WBS

WBSは、Work Breakdown Structureの略で、プロジェクトに必要な作業を階層的に分解したものである。

メリット:

- スコープが明確になる
- 必要な作業を洗い出せる
- やらないことも明確になる
- 全体管理と作業計画がしやすくなる

## WBSの作り方

WBSは以下の流れで作る。

1. スコープを明確にする
2. 大きな作業単位に分ける
3. 作業同士の関係を考える
4. 各作業を細かく洗い出す
5. 担当者と期間を決める

作業を洗い出すときは、MECE、つまり漏れなく・ダブりなく整理することが重要である。

## WBS演習

演習では、情報工学科の学生が参加するゲーム大会を、3人で企画・運営する想定でWBSを作成する。

必要な内容:

- スコープ定義
- 想定条件
- フェーズ設定
- 全体WBS
- 当日WBS
- 役割分担
- 各タスクの開始日・終了日

当日の進行、トラブル対応、片付けまで含めて考える必要がある。

## 読みやすいコード

コードは書かれる回数よりも、他人や未来の自分に読まれる回数の方が多い。

そのため、ソフトウェア開発では、統一されたルールに従って読みやすいコードを書くことが重要である。

主な目的:

- コードを理解しやすくする
- 保守や修正をしやすくする
- チーム内で書き方を統一する

コーディング規約には、PythonのPEP 8やGoogle Style Guideなどがある。

## PEP 8

PEP 8は、Pythonの代表的なコーディング規約である。

主なルール:

- 1行は原則79文字以内にする
- コメントやdocstringは72文字以内にする
- インデントにはタブではなくスペース4つを使う
- 演算子の前後には適切にスペースを入れる
- 不要なスペースを入れない
- 複数の処理を無理に1行へまとめない
- 長い式を改行するときは位置を揃える

規約に従うことも重要だが、プロジェクト全体で一貫した書き方をすることがより重要である。

## importと命名規則

importは、以下の順番でグループ分けする。

1. 標準ライブラリ
2. サードパーティ製ライブラリ
3. ローカルのアプリケーションやライブラリ

代表的な命名規則:

- パッケージ: 小文字
- モジュール: 小文字、必要に応じてアンダースコアを使う
- クラス・例外: `CapWords`
- 関数・メソッド・変数: `snake_case`
- 定数: `UPPER_CASE`

`l`、`O`、`I`のように数字と見分けづらい1文字の変数名は避ける。

## Linter

Linterは、コードの問題やコーディング規約違反を自動で検出するツールである。

Pythonの代表例:

- flake8
- pylint

flake8では、主に以下の種類の問題が表示される。

- `F`: コード上のエラー
- `E`: スタイル違反
- `W`: スタイルに関する警告

Linterを利用することで、コードレビュー前に問題を発見し、コード品質を一定に保ちやすくなる。

## Git

Gitは、ソースコードの変更履歴を記録・管理するための分散型バージョン管理システムである。

変更をコミットとして保存することで、過去の状態を確認したり、複数人で並行して開発したりできる。

基本的な流れ:

1. ファイルを変更する
2. `git add`で変更をステージングする
3. `git commit`で変更履歴を保存する
4. `git push`でリモートリポジトリへ反映する

## 主なGitコマンド

設定・確認:

- `git init`: Gitリポジトリを初期化する
- `git status`: ファイルの状態を確認する
- `git config`: Gitの設定を確認・変更する
- `git log`: コミット履歴を表示する
- `git diff`: ファイルの差分を表示する

コミット:

- `git add`: 変更をステージングエリアへ追加する
- `git commit`: 変更をコミットとして記録する
- `git commit --amend`: 直前のコミットを修正する

変更の取り消し:

- `git reset`: コミットやステージング状態を戻す
- `git revert`: 過去の変更を打ち消す新しいコミットを作成する
- `git rm`: ファイルとGit上の管理情報を削除する

## ブランチ

ブランチは、変更履歴を分岐させて作業するための機能である。

新機能の開発やバグ修正を別のブランチで行うことで、メインのコードへ影響を与えずに作業できる。

主なコマンド:

- `git branch`: ブランチを作成・確認する
- `git checkout`: 使用するブランチを切り替える
- `git merge`: 別のブランチの変更を統合する

複数人が別々のブランチで作業し、完成後に変更を統合することで、並行開発がしやすくなる。

## リモートリポジトリ

リモートリポジトリは、ネットワーク上で共有されるGitリポジトリである。

主なコマンド:

- `git clone`: リモートリポジトリをローカルへ複製する
- `git pull`: リモートの変更を取得して反映する
- `git push`: ローカルの変更をリモートへ送信する
- `git remote`: 接続するリモートリポジトリを設定・確認する

ローカルリポジトリとリモートリポジトリを同期することで、複数人が変更を共有できる。

## GitHub

GitHubは、Gitのリモートリポジトリを提供するホスティングサービスである。

ソースコードを共有するだけでなく、開発作業や課題を管理するための機能も提供している。

主な機能:

- リポジトリの共有
- Issueによる課題管理
- Projectsによるタスク管理
- Pull Requestによる変更の提案・確認
- オープンソースソフトウェアの公開・共同開発

## clone・fork・template

`clone`は、リモートリポジトリをローカル環境へ複製する機能である。

`fork`は、他のユーザーのリポジトリを自分のGitHubアカウントへ複製する機能である。元のリポジトリとは独立して変更できる。

テンプレートリポジトリは、共通の構成をもとに新しいリポジトリを作成するために使われる。

## IssueとProjects

Issueは、リポジトリに関する以下の情報を管理する機能である。

- バグ
- 課題
- 機能追加
- 改善案
- 質問

Projectsは、Issueなどを表やカンバン形式で整理し、進捗を管理する機能である。

## GitHubフロー

GitHubフローは、ブランチとPull Requestを利用した開発方法である。

基本的な流れ:

1. mainブランチから作業用ブランチを作成する
2. 作業用ブランチで変更する
3. 変更をコミットする
4. ブランチをGitHubへpushする
5. Pull Requestを作成する
6. 変更内容を確認・レビューする
7. mainブランチへマージする

変更を直接mainブランチへ追加しないことで、安全に開発やレビューを行える。

## MarkdownとCodespaces

Markdownは、簡単な記法で文章を構造化できる形式である。

主な特徴:

- 拡張子は`.md`
- 見出し、一覧、リンクなどを簡単に記述できる
- GitHubを含む多くのサービスで利用されている
- HTML、PDF、Wordなどへ変換できる

GitHub Codespacesは、ブラウザ上にVisual Studio Code形式の開発環境を作成できるサービスである。

ローカル環境を構築しなくても、リポジトリの編集、Git操作、プログラムの実行などを行える。

## CI/CD

CI/CDは、コードの統合、テスト、ビルド、デプロイを自動化する仕組みである。

### CI（Continuous Integration）

CIは、コードの変更を共有リポジトリへ頻繁に統合する考え方である。

自動的にビルドやテストを実行することで、問題を早い段階で発見・修正できる。

主な目的:

- バグの早期発見
- コード品質の維持
- 統合作業の効率化

### CD（Continuous Delivery）

CDは、テスト済みのコードをテスト環境や本番環境へ継続的に提供する仕組みである。

デプロイ作業を自動化することで、変更を早く安全にユーザーへ届けられる。

## CI/CDパイプライン

一般的なCI/CDパイプラインは、以下の流れで実行される。

1. ソース: コード変更をきっかけに処理を開始する
2. ビルド: ソースコードを実行可能な状態にする
3. テスト: 自動テストを実行する
4. デプロイ: テスト済みのコードを環境へ配置する
5. 検証: デプロイ後の動作を確認する
6. モニタリング: 本番環境を継続的に監視する

問題が発見された場合は修正を行い、再びパイプラインを実行する。

## GitHub Actions

GitHub Actionsは、GitHub上でCI/CDなどの処理を自動化する機能である。

ワークフローはYAML形式で記述し、`.github/workflows`ディレクトリへ配置する。

主な構成:

- ワークフロー: 自動化する処理全体
- トリガー: ワークフローを開始する条件
- ジョブ: 実行する処理のまとまり
- ステップ: ジョブ内の個別の処理
- アクション: 再利用可能な処理
- ランナー: ワークフローを実行する環境

トリガーには、push、Pull Request、定期実行などを設定できる。

## GitHub Actionsによる自動チェック

GitHub Actionsとflake8を組み合わせることで、Pull Request作成時などにPythonコードを自動で検査できる。

処理の例:

1. Pull Requestを作成する
2. GitHub Actionsを自動で起動する
3. Python環境を構築する
4. flake8を実行する
5. コードの問題を検出する
6. 検査結果を確認する

CIによって、問題のあるコードがmainブランチへ統合される前に、品質を確認できる。
